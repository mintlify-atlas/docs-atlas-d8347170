## HAL/HATEOAS

Every resource response from the Spring Data REST layer includes a `_links` object containing URIs for the resource itself and related resources. This follows the [HAL (Hypertext Application Language)](https://stateless.group/hal_specification.html) standard.

Example response for `GET /editions`:

```json
{
  "_embedded": {
    "editions": [
      {
        "year": 2025,
        "venueName": "Barcelona",
        "description": "FLL Season 2025",
        "state": "DRAFT",
        "_links": {
          "self": { "href": "http://localhost:8080/editions/1" },
          "edition": { "href": "http://localhost:8080/editions/1" }
        }
      }
    ]
  },
  "_links": {
    "self": { "href": "http://localhost:8080/editions" }
  },
  "page": {
    "size": 20,
    "totalElements": 1,
    "totalPages": 1,
    "number": 0
  }
}
```

## Spring Data REST endpoints

All repository-backed resources are auto-exposed by Spring Data REST at their canonical collection path. Standard CRUD operations are available for each resource.

| ---- | ----------- |
| `/users` | User accounts |
| `/teams` | Competing teams |

## Pagination

Collection endpoints support pagination through query parameters:

| Parameter | Default | Description |
| --------- | ------- | ----------- |
| `page` | `0` | Zero-based page number |
| `size` | `20` | Number of items per page |
| `sort` | — | Field to sort by, e.g. `sort=year,desc` |

Example:

```bash
curl http://localhost:8080/editions?page=0&size=10&sort=year,desc
```

## Custom endpoints

Beyond the auto-generated CRUD endpoints, the API exposes custom controllers that implement business logic:

| Endpoint | Description |
| -------- | ----------- |
| `PATCH /editions/{editionId}/state` | Transition an edition through its lifecycle states |
| `POST /editions/{editionId}/teams/{teamId}` | Register a team to an edition |
| `GET /editions/{editionId}/volunteers` | Retrieve edition volunteers grouped by role |
| `GET /editions/{editionId}/tables` | Retrieve competition tables with scheduled matches |

See the following pages for full documentation of each custom endpoint.

## Error response format

All error responses from custom endpoints use a common `ApiErrorResponse` structure:

```json
{
  "error": "EDITION_NOT_FOUND",
  "message": "Edition with id 42 not found",
  "timestamp": "2025-03-15T10:30:00.000Z",
  "path": "/editions/42/state"
}
```

The `details` field is included only when additional structured context is available (for example, validation errors).

<ResponseField name="error" type="string" required>
  Machine-readable error code identifying the failure type.
</ResponseField>

<ResponseField name="message" type="string" required>
  Human-readable description of what went wrong.
</ResponseField>

<ResponseField name="timestamp" type="string" required>
  ISO-8601 UTC timestamp of when the error occurred.
</ResponseField>

<ResponseField name="path" type="string" required>
  Request URI that produced the error.
</ResponseField>

<ResponseField name="details" type="object">
  Additional structured error context. Omitted when not applicable.
</ResponseField>

## HTTP status codes

| Status | Meaning |
| ------ | ------- |
| `200 OK` | Request succeeded. |
| `201 Created` | Resource was created successfully. |
| `400 Bad Request` | Request body is missing, malformed, or contains an invalid value. |
| `401 Unauthorized` | Authentication is required but was not provided. |
| `403 Forbidden` | The authenticated user does not have permission to perform the action. |
| `404 Not Found` | The requested resource does not exist. |
| `409 Conflict` | The request conflicts with the current state of a resource (e.g., invalid state transition, duplicate registration). |
| `422 Unprocessable Entity` | The request is well-formed but cannot be executed due to business rule constraints. |
